基础原语审查生死线 (Primitive Review Cycle)

状态： 方法论指导书。这篇文档讲的是：在把任何一个新的 BENE 运行时原语 (runtime primitive) 死死焊进边界计划 (boundary plan) 之前，到底该怎么去验它的成色。

配套读物： philosophy.md (扒一扒为什么 BENE 把身家性命全押在引擎的进化上), codegen.md (把 IR 当成代码生成靶心的终极愿景), 以及真刀真枪的运行时边界代码 bene/runtime/handle.py (协议) + bene/runtime/core.py (DTO)。

这套繁文缛节到底是图什么

当 BENE 考虑往运行时边界里加点新花样时 (比如给 RunHandle 挂个新方法，给命令包塞个新字段，或者挂个新的生命周期钩子)，无论哪个评审员来审，这套设计都绝对会带上至少三个盲区 —— 这是单兵作战绝对跨不过去的槛：

  1. 原作者的盲区 (The owner blind spot)。 捏出这个抽象概念的人，看问题的视角永远是从 "怎么实现它" 出发的。他们知道这玩意 "能造得出来"，但他们绝不会闲得蛋疼去把这套玩意套在业务代码里，去跑跑重试 (retry)、重放 (replay) 或者半路炸机 (partial failure) 的真实场景。这帮人极度容易吹嘘自己的原子性 (atomicity) 保证，同时又把业务方为了用它而被迫写的胶水代码 (boilerplate) 描述得轻描淡写。
  2. 第一口吃螃蟹者的盲区 (The consumer blind spot)。 第一个把这玩意用到生产环境的人，满眼全是自己那点破业务。他们根本看不出这套 API 稍微换个场景就会水土不服，看不出那些只有等 LocalRuntime 也要实现同一套协议时才会引爆的 DTO 乱放问题，更想不到那些他们自己碰巧没用上的外部系统，到底该怎么做状态对齐 (reconciliation)。
  3. 吃瓜主持人的盲区 (The facilitator blind spot)。 那个在中间跑腿传话、攒局讨论的人 (通常是旁观这两头吵架的第三方)，既没有原作者那种对底层实现的直觉，也没有第一线业务方那种在生产环境里被毒打过的经验。这帮人最喜欢发明那种听起来极度优雅、但只要一接触真实的实现细节，或者一遇上真实的生产环境重试风暴，就会瞬间被撕成碎片的 "完美抽象"。

如果只找一个人来审设计，他绝对会精准地漏掉三个盲区里的两个。哪怕你把作者和业务方两个人拉到一个屋子里吵架，他们依然会漏掉第三方的那个终极拷问："你这套狗屁抽象，到底是真的帮业务方兜了底，还是只是把一堆恶心的复杂性换了个地方藏起来而已？"

这套基础原语审查生死线 (Primitive Review Cycle)，就是强制拉来三个活人 —— 死死钉在上面那三个角色上 —— 然后逼着这套设计在三个绞肉机循环里滚一遍。每个循环的目的，就是为了把某种特定的 bug 从盲区里逼出来。

绞肉机三部曲

Cycle 1: 嘴炮局 (Consultative review)

对业务方的拷问: "这是我们新想出来的原语草图。你把它往你那真实的业务逻辑里套一套，严丝合缝吗？你要是能当家，你想改什么？"

这轮能逼出什么病: 概念上的鸡同鸭讲；那些只有当业务方真的去脑补自己的代码时，才会发现漏掉的致命参数；以及极其宝贵的、帮业务方减负的机会 (一线干活的人，往往能想出比原作者更接地气的调包姿势)。

战利品: 一份吸收了业务方吐槽后修改过的原语草图。这时候还没开始写代码。

这轮绝对抓不到的病: 原作者在吹牛逼 (运行时到底能给多大保证，根本没法在这轮查实)。半路炸机时的惨状。跨业务方的 DTO 冲突。这轮的业务方只是在脑补一张草图，还没真正上刑具。

Cycle 2: 代码生成压力测试 (Codegen stress test)

给业务方派活: "拿着刚刚那套草图当真家伙用，给老子直接生成一套能上生产环境的、带全套业务逻辑的代码出来。测试用例必须全覆盖，重试、重放、半路炸机，一个都不能少。"

这轮能逼出什么病: 任何业务方需要分多步执行的操作，瞬间就会暴露出来的 TOCTOU (Time-of-check to time-of-use) 竞态灾难；业务方为了填坑，被迫在现场徒手捏出来的不存在的原语；纸上谈兵的草图和真刀真枪的契约之间的巨大鸿沟；以及那些在伪代码里看着极度优雅，一敲到 IDE 里就恶心到吐的 API 形状。

战利品: 活生生的代码，直接提交到一个分支上。活生生的测试用例。一份极其羞耻的自白书：业务方必须交代，在写这套代码时，到底有几次被逼得没办法，强行绕开了这套抽象。

这轮绝对抓不到的病: 这玩意到底有多难实现。原作者关于外部系统原子性的吹牛逼。会把其他运行时一起坑死的模块放置问题。业务方只是对着某一个运行时底座 (通常是离他们业务最近的那个) 狂敲代码，他们根本不会去操心整套协议在全局视角的死活。

Cycle 3: 原作者的终极审判 (Owner ratification)

给原作者派活: "拿着业务方敲出来的那堆破铜烂铁，重新审视你设计的原语。给我死死咬住这三点往死里喷：(a) 运行时到底能不能兑现你吹出去的那些保证，(b) 这些 DTO 放在这，会不会把其他运行时给坑死，(c) 这套抽象的接口表面，以后升级版本时会不会要了老命。对所有遗留的 related todos 追溯执行 Design Review Gates。"

这轮能逼出什么病: 被戳穿的原子性吹牛逼 (运行时大包大揽了只有外部系统才能兑现的承诺)；极度脑残的模块放置 (比如逼着 LocalRuntime 去依赖 TemporalRuntime 里的模块)；外部写入时缺失的状态对齐 (reconciliation) 策略；以后升版本时的死胡同；以及那些绝对会把以后的维护者看懵逼的狗屎文档。

战利品: 最终版的基础原语规格说明书 —— 带着极其诚实的原子性边界说明、极其干净的 DTO 模块归属、极其硬核的状态对齐策略、并且把所有 related todos 的 Design Review Gates 全填得清清楚楚。

这轮绝对抓不到的病: 第二个吃螃蟹者会带来的全新需求。这得靠活生生的第二个业务方来验货，这就是判断这套原语到底能不能泛化铺开的唯一标准 —— 请看下面的 "什么时候该重开一轮"。

案发现场复盘：submit_side_effect 的诞生 (2026-05-04)

这个套路，是在我们为 BENE 运行时边界计划去死磕四个新原语 (start_or_signal, business_idempotency_key, side_effect_label，以及最后被逼出来的 submit_side_effect) 时摸索出来的。这个摸索的过程本身，就是一场极度硬核的 Primitive Review Cycle 实战。

Cycle 1 的下场。 第一口吃螃蟹者 (l3 突发警报聚合器 session) 孤立地审了前三个原语。判决：全盘接受，并且反手塞给原作者一个极其刁钻的优化 —— 直接拿 int(timestamp / delta_burst) 去当工作流的 ID，这样从根本上就把 TTL 的语义锁死在了滑动时间窗 (tumbling-window) 上，并且天然地卡死了工作流的历史记录膨胀。这个极其优雅的套路被当场提拔进了边界计划里，成了时间窗实体 Actor (time-bounded entity actors) 的官方推荐写法。

Cycle 2 的下场。 业务方被逼着拿着那几个还在纸上的原语，去生敲真刀真枪的代码 (burst_aggregator.py，339 行代码 + 419 行测试)。业务方提着带血的代码回来报案：你们设计的这堆破原语里，压根就没给外部写入 (在这个场景下是对 Jira 发 POST) 最需要的那个 "检查-然后-写入" (check-then-write) 动作提供一个原子操作。业务方被逼得没办法，只能在现场徒手捏出了 runtime.check_side_effect 和 runtime.record_side_effect。在这两个调用之间，只要碰上 worker 炸机重启，Activity 一重试，外部写入就会再跑一次 —— 生产环境的 Jira 里直接喜提两条重复评论。业务方拍桌子要求：加一个 submit_side_effect(label, business_key, execute_fn) -> result，把这个动作全包在一个原子动词里，让运行时自己去把锁给捏死。如果光靠 Cycle 1 里对着草图打嘴炮，这个致命的空门这辈子都发现不了。

Cycle 3 的下场。 原作者 (bene 边界计划 session) 捏着鼻子认了业务方提出的 submit_side_effect 原语，但在它的语义上极其霸道地怼了回去。运行时根本就没那个本事去保证外部系统 (比如 Jira) 的跨库原子性 —— 它只能管好自己的一亩三分地和租约。所以必须把牛逼收一收，诚实的契约是：运行时能死保账本流转的原子性 (pending → executing → done | failed) 和租约的独占性；但对于外部写入，那个 SideEffectLabel 必须老老实实声明一套自己的状态对齐策略 (reconciliation_strategy，比如 IDEMPOTENT_BY_KEY, READBACK_SENTINEL, READBACK_RESULT_KEY, 或 MANUAL)。当系统重试时，靠着这套策略去查前一次写入到底落地了没有。不仅如此，原作者还抓包指出：新加的这几个公共 DTO 绝不能塞在 bene/temporal/ 底下，因为 LocalRuntime 也要实现这套协议 —— 它们必须滚去像 bene/runtime/core/ 这种跟底层实现无关的公立地盘。最后，原作者极其负责地给三个遗留的 related todos 补上了 Design Review Gates。

Cycle 3 后业务方的补刀。 拿着这套被精准阉割过、语义极度清晰的 submit_side_effect (外加原作者建议 Jira 使用 READBACK_SENTINEL，也就是插个形如 <!-- bene-business-key: {key} --> 的暗号)，业务方重新去翻修了生成的代码和测试。在这波翻修里，业务方又踩中了一颗极其隐蔽的现实地雷：Jira 的 ADF 格式会在 JSON 序列化时把这个 HTML 暗号强行包在一个段落节点 (paragraph node) 里，而不是老老实实当成个原生 HTML 注释。这意味着运行时的状态对齐扫描，必须去查那坨恶心的 JSON 格式，而不是去查渲染出来的 HTML。这个雷其实在业务方自己的代码库 (check_existing_advisory.py:76-84) 里早就有前科了，属于这个领域的深坑 —— 但这也就意味着，READBACK_SENTINEL 作为一套通用策略，必须得开个口子，允许外部系统自己去指定 "去哪里扫暗号"。这件事被记进了后续的账本里，但不卡这次原语的落地。

跑完一整个 Cycle 的战果： 四条新加进边界计划的待办清单 (有一条是死线 BLOCKING 级别的)；三个被填上的 Design Review Gates；一个被标记出来的未来填坑计划 (READBACK_SENTINEL 的扫描位置参数)；以及分支上一套 339+419 行的标杆级实盘代码。如果只是找一个人随便审审，这些战果一个都捞不到。

这套机制为什么能赢

这套绞肉机之所以极其有效，是因为这三个评审员在结构上有着根本互斥的盲区，而整个绞肉机的流程就是为了在不同的阶段把每个盲区逼到死角里。

原作者根本看不出业务代码里会有 TOCTOU 竞态，因为原作者压根就不写业务代码。这套机制直接让 Cycle 2 逼着业务方在绝境中敲出实盘代码，把这个雷给引爆。

业务方根本看不出原子性吹牛逼的风险，因为在业务方的脑子里，运行时就是一个黑盒 —— 只要 API 敢写 "原子操作"，业务方就敢闭着眼睛信。这套机制直接让 Cycle 3 逼着原作者去直面业务方的压力测试产物，自己把吹破的牛皮圆回来。

主持人 (通常是那个在两边跑腿、并且最后把结论写进计划书里的人) 根本看不出实现成本的代价，因为写代码擦屁股的不是他。这套机制直接让原作者去审核 (或者暴力修改) 所有主持人提出来的漂亮空话，确认没坑了，才准写进计划书里。

每个循环的残骸，就是下一个循环的炮弹。当三轮走完时，这个原语已经被从三个完全不搭界的角度拿重火力犁了一遍 —— 活下来的那份规格书，就是见识过地狱的铁证。

什么时候该用这把杀猪刀 (什么时候千万别用)

什么时候该上大刑: 只要是准备往运行时边界上加新原语；给共享的命令 DTO 塞新字段；加新的生命周期钩子；拍板新的模块放置策略；或者任何一旦落地，就会坑死无数业务方、以后想改都难如登天的大手术。

什么时候别吃饱了撑的: 修个破 bug；没动边界的内部重构；只有一个人在用、并且明确声明是随时扔掉的实验品；或者设计本身已经被一个根本改不了的外部 API (比如你要去接某个第三方的 SDK) 给锁死了的时候。

这套绞肉机的成本极高 —— 它会活生生吃掉三个 session 的时间，外加主持人的各种跑腿调度。只有当 "设计翻车带来的灾难" 远远盖过 "跑一轮 Cycle 的成本" 时，才值得拔出这把刀。对于那种在边界上被 N 个团队用上好几年的基础原语，这点测试成本连根毛都算不上。但如果你只是在写一个只有你自己会调用的内部小脚本，用这套方法纯属杀鸡用牛刀。

具体该怎么操办

拿着 2026-05-04 那个案发现场的实盘来做个示范：

起手式。 主持人 (这次是个 cursor session) 去把现有的计划书看透，把那些新提出来的原语圈出来，然后给业务方 session 准备一份极度聚焦的战书。战书里必须有：原语草图、3 到 5 个逼着业务方必须正面回答的死命题、以及业务方需要的任何背景文档。

Cycle 1 发车。 主持人通过 tmux paste-buffer 把战书拍在业务方的脸上。业务方不用切环境，就在自己手头的环境里直接回话。业务方把那几个命题一一作答，顺便丢出自己的修改意见。

Cycle 2 发车。 主持人拿着 Cycle 1 收集来的口水修改原语草图，然后反手甩给业务方一个大招：给我生敲一套真家伙出来。把 IR 目标文档和规格说明书拍给业务方，命令他们在一个干净的分支里，把代码和测试全敲出来。业务方敲完后，还得对着主持人列出的及格线自己做个自我检讨。

Cycle 3 发车。 主持人拿着业务方敲出来的那堆带着血的代码、以及 Cycle 2 里业务方在现场无中生有捏出来的不存在的原语，跑去跟原作者 session 告状。原作者拍板、把那些吹过头的承诺砍掉、把相关遗留的 todos 全拿 Design Review Gates 犁一遍，并且 (非常频繁地) 顺手把主持人漏看的底层架构雷给排了。

Cycle 3 的回马枪。 主持人拿着 Cycle 3 里被精准阉割过的契约，跑回去通知业务方修改代码。在这个翻修的过程中，业务方极其容易踩中一些极其隐蔽的现实地雷，这些地雷就成了以后的填坑计划。

计划落地 (Plan commit)。 主持人把这几轮里吵出来的所有结果 (新的 todos、修改过的老 todos、边界改动、Design Review Gates、以及案发现场复盘) 一字不差地写进计划书里，提交。

把这三轮全部走完，这三个 session 加起来撑死也就花个两到四个小时 (现实世界的时间)。这其中绝大多数时间全是业务方在 Cycle 2 里生敲代码 —— Cycle 1 和 Cycle 3 这种打嘴炮的局，每个 session 也就是几分钟到十几分钟的事。

还有什么坑没填

这套方法论以后还可以考虑往这些方向继续榨汁：

  • 抓两个小白鼠来试毒 (Two-consumer validation)。 现在文档里写的这套 Primitive Review Cycle，从头到尾只有一个业务方在被来回折腾。更变态的测法是：等第一个业务方把这三轮地狱全走完了，再抓第二个毫不相干的业务方，让他只走一遍 Cycle 1 —— 看看这套原语在第二个业务方眼里是不是依然极度优雅，还是说它已经完全被第一个业务方给带跑偏了，成了一个高度定制的奇葩？还没想明白的事：到底该在什么时候把第二个小白鼠放进来？放早了，整个 Cycle 会被搞得乌烟瘴气；放晚了，被带跑偏的雷可能已经埋下了。
  • 原作者的自我审判 (Owner self-cycle)。 当这套原语压根就没有业务方，原作者就是唯一的用户时 (原语刚起步时的常态)，这套绞肉机就会瞬间坍缩，那个靠角色互斥来扫盲区的核心机制直接报废。还没想明白的事：当只有一个人时，有没有什么变种的玩法能稍微找回一点扫盲区的感觉？还是说这种活必须得有多人互砍？
  • 杀鸡用牛刀的简化版 (Cycle compression for small primitives)。 只是给命令加个破字段，真犯不上拿这套大刑伺候。还没想明白的事：怎么在这个大刑里抽掉一些极其繁琐的仪式，但同时又能死死保住 "盲区轮换" 这个核心灵魂？

等哪天我们又要往 BENE 里塞新原语了 (比如那个传说中的第二个实体 Actor 终于落地，又要被拉出来用这套方法毒打时)，这些没想明白的事就可以拉出来练练了。

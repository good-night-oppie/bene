## 鸟瞰 (Overview)

这八块拼图拼完，BENE-v0.2 内核的底盘将从 "管用但被动" (印记、探针、信用、ContextOS 都搞定了，但得靠人拨一下才动)，彻底进化成 "全自动活物" (拉磨的调度器、能签字画押的重放机、会自我修剪废柴技能的神经可塑性、看熵下菜碟的路由机制，甚至能在干活中途直接变异的引擎)。有 5 个特性严丝合缝地扣进了现有内核边界里；另外 3 个在红蓝对抗里被揪出了核心破绽，必须重做 —— 详情看每节的 **对抗裁决 (Verdict)**。

**打死不能破的铁律 (在所有设计里通用)：**

- 授予 L4 级兵器必须有 `granted_by="human:<name>"` 的玉玺印记 —— 任何图纸都不准搞 "自动升级 L4" 的鬼把戏。
- 探针的死门禁必须是被 sha256 锁死的；敢动探针规则一个字，当场 REJECT 斩首。
- 过死门禁只有 `promote()` 这一条活路；绝对不允许留后门开小灶。
- 必须有血脉来源 (provenance)；不管是晋升还是降级，永远只能在档案上新盖一个戳，**绝对不准动原版数据的任何一根汗毛**。

## 1. `bene replay` — 带签名的确定性重放 CLI

**状态 (Status)**: 半成品 (partial)。流水账大巴 (Journal) 和快照 (checkpoints) 底盘早就做好了；定时梦境整合 (scheduled consolidation) 现在也能吐出带确定性算力分配清单 (deterministic replay manifests) 的 `experiment_runs` 行。这部分已经 **实锤验证过 (VERIFIED)** (去查 `tests/kernel/test_memory_os.py::test_scheduled_consolidation_plans_replay_manifest_deterministically`)。
现在缺的是：(a) 一个 CLI 工具，能把这份清单拿去沙盒库里重新跑一遍，并拍着胸脯保证跑出来的印记一个字节都不差；(b) 一个带 ed25519 签名的加密信封；(c) 吐出一份 BibTeX 的学术引用，方便像 `adx-cli-7` 这种老外团队拿去做 11d 槽位的对抗见证。
**代码量估算**: 约 420 行
**对抗裁决**: ✅ 无懈可击 (clean) (四个维度全部过关)

### 现场还原 (Scene)
凌晨 04:32，你正在跟 `adx-cli-7` pane 喊话，他们那个 11d.4 PR 非要把 BENE 的梦境整合拉过去当 adversarial witness 引用——他们打开 `experiment_runs` 看到一串 ULID 和一个看不懂的 metrics blob，发难说："这玩意我怎么 re-run？我怎么知道你没偷偷改数据？"
你直接反手敲个 `bene replay export <run_id> --sign`，丢过去一个 `.bene-replay` 文件。对面在自己机器上跑一句 `bene replay verify <file>`，MD5 严丝合缝对上，BibTeX 字段直接 copy-paste 进他们的 paper draft。闭嘴干活。

### 痛点 (Problem)
今天，`experiment_runs` 里记了一笔 "我们跑过一次梦境整合"，但是下游的消费者 (`adx-cli-7`、论文评审、明天的你) 没法顺着那几个碎片渣子重新把印记还原出来，也防不住别人篡改清单。没有这个带签名的 `bene replay` 动作，你们跨团队的每次引用，说白了都是在刷脸（"trust-me link"）。

### 兵器长什么样 (Proposed API)

```python
# bene/kernel/replay/manifest.py
@dataclass(frozen=True)
class ReplayEnvelope:
    version: int                 # 版本号，目前是 1
    run_id: str
    kind: str                    # 今天只有 "consolidation"；以后会有 "probe"/"evolution"
    manifest: dict               # 一字不差的 ConsolidationPlan.replay_manifest() 输出
    source_payload_hashes: dict[str, str]  # engram_id -> sha256(payload bytes) 的指纹图谱
    bene_version: str            # 取 pyproject 里的 __version__
    schema_version: int          # 取 kernel_schema_version
    created_at: str
    signature: str | None        # base64(ed25519) 签名；如果不带 --sign 就不签
    public_key: str | None       # base64 公钥

class ReplayExporter:
    def __init__(self, store: EngramStore): ...
    def export(self, run_id: str, *, sign_key: Ed25519PrivateKey | None) -> ReplayEnvelope: ...

class ReplayVerifier:
    def verify(self, env: ReplayEnvelope, *, into_db: Path) -> ReplayResult:
        """拿着指纹图谱在沙盒库里重新开天辟地，然后卡死这四条命脉：
        (1) 签名不能破，(2) 重新演算的清单必须 == 信封里的清单，
        (3) 源文件的哈希指纹必须全对上，(4) 如果是探针任务，lock_sha256 敢变一下直接砍。"""

@dataclass(frozen=True)
class ReplayResult:
    ok: bool
    reasons: tuple[str, ...]     # 只有 ok=True 时才为空
    replayed_manifest: dict
    cite_as: str                 # BibTeX 引用格式 @misc{bene:<run_id>, ...}
```

```bash
# 命令行武器库 (全在原来的基础上加料，不另起炉灶)
bene replay export <run_id> --out run.bene-replay [--sign] [--key-file ed25519.key]
bene replay verify run.bene-replay [--into /tmp/replay.db] [--json]
bene replay cite   <run_id> [--style bibtex|json]    # 不想导出文件，只想掏个引用字符串
bene replay ls                                       # 看看有哪些能导出的 run_ids (去查 experiment_runs 表)
```

裸敲一句 `bene replay <run_id>` (不带子命令)，就是在活库上就地跑个 `verify` 的快捷键 (纯内存里重演一遍，断言全等，敢有一点差错直接返回非零退出码)。

### 数据长什么样 (Data structures)
V1 版本**一行新表都不建**。就加了两个极小的东西：

1. **印记种类 (kind) 偷偷蹭车** (不需要改表结构，直接写在 `metadata` JSON 里)：当 `bene replay export` 跑完时，它会顺手造一个 `kind='report'` 的印记，标题写着 "replay-envelope:<run_id>"。把极度标准的信封 JSON 塞进 `inline_body` 里，再打上防伪戳 `provenance={"system":"bene.kernel.replay","source_run":run_id}`。这样，信封本身也变成了一条堂堂正正的、能被引用的印记 (下游在 `engram_links` 里拉线时就能正大光明地说 "这篇 paper 引用了那次 replay")。
2. **`bene.yaml` 里加个选配块**:
   ```yaml
   kernel:
     replay:
       sign: true
       key_file: ~/.config/bene/replay_ed25519.key   # 这玩意是 `bene replay keygen` 搓出来的
       trusted_keys: ~/.config/bene/replay_keys.json
   ```
   顺着原有的 `bene/config.py` 经脉加载；如果没这段，就允许不签名导出；不带签名的信封也能跑 verify，只是 `ReplayResult` 会在脸上给你糊一个 `signature="absent"`。

至于专用的 `replay_runs` 表，等以后第二波 (探针/进化) 上来再说 —— 现在的 `experiment_runs` 足以兜底。

### 怎么落地 (Implementation sketch)
**责任人**: 在内核开辟一块新辖区 `bene/kernel/replay/` (manifest.py, exporter.py, verifier.py, cite.py, keys.py)。这块辖区必须**老老实实呆在印记阶梯 (engram-ladder) 的羽翼下** —— 它就是一个趴在 `experiment_runs` 和它引用的印记上喝风的只读投影机，别妄想搞成什么独立新山头。

**数据管线 (Dataflow)**:
1. `export(run_id)` 把 `experiment_runs` 里的数据扒出来。里面的 `metrics` 本来就是 `json.dumps(..., sort_keys=True)` 锁死过顺序的。然后顺着清单里的 `source_ids` 把碎片肉块掏出来，算一遍 `sha256(payload)`。最后用 `cryptography.hazmat.primitives.asymmetric.ed25519` 把整个压扁的 JSON (键排好序，去空格) 签个字 (包已经在 uv.lock 里了，不用加新依赖)。
2. `verify(env, into_db)` 在沙盒里支一口 sqlite 临时锅，跑个 `ensure_v2`。照着指纹图谱捏一堆影子印记塞进去。把 `ScheduledConsolidator` 喊出来，逼它照着信封里的规矩再跑一遍 `.plan(policy=..., force=True)`，死死盯住它：`plan.replay_manifest() == env.manifest`。顺手把公钥签名验了。
3. `cite_as` 就是照着信封印一套名片。
4. CLI 界面全挂在 `bene/cli/main.py` 的 `@cli.group("replay")` 下面，复用现成的 `_kernel_stores(db)` 连接池和 `--json` 管道。

**怎么跟 5 大台柱子咬合**:
- **印记阶梯 (Engram ladder)**: 纯只读的吸血鬼；绝对不准改源文件 (死守血脉来源铁律)。
- **探针与死门禁 (Probe + kill gate)**: 如果信封里装的是 `kind="probe"`，信封会自动把探针当时的 `lock_sha256` 钉死在上面。在 verify 的时候，活库里的探针只要敢动过一个字节导致 sha256 对不上，`ReplayResult.ok` 当场炸成 False，并甩出一句 "probe-spec-mutated" (探针被掉包)。这其实就是现有的自动斩首门禁套了层重放检验的壳，不是新加的规矩。
- **Context OS / 兵权 / 进化大管家**: 不挨边。

### 怎么测 (Eval plan)
探针 A — `replay.roundtrip.consolidation` (只认死理：导出的和验出来的一字不差就算 ACCEPT):
```bash
uv run bene replay export $(uv run --quiet bene --json experiments ls --kind consolidation | jq -r '.[0].run_id') --out /tmp/r.bene-replay --sign && uv run bene replay verify /tmp/r.bene-replay --json | jq -e '.ok == true and .reasons == []'
```
ACCEPT 条件：退出码为 0，`.ok == true`，重新演算的清单跟信封里的字节级一模一样。REJECT 斩首：只要有一个条件不满足。这句破命令将被死死地钉在探针的 sha256 锁上。

探针 B — `replay.tamper.detect` (抓贼模式，揪出动过哪怕一个字节的信封才算 ACCEPT):
```bash
uv run bene replay export <run_id> --out /tmp/r.bene-replay --sign && python -c "import json,sys; p=open('/tmp/r.bene-replay','r+'); d=json.load(p); d['manifest']['batches'][0]['summary']+=' '; p.seek(0); p.truncate(); json.dump(d,p)" && uv run bene replay verify /tmp/r.bene-replay --json | jq -e '.ok == false and (.reasons | index("signature-invalid"))'
```
ACCEPT 条件：退出码 0，而且 `.ok == false` 加上被扇巴掌的理由里带着 `signature-invalid`。REJECT 斩首：放过了贼就算你死。这是探针 A 的红蓝对抗双生子。

### 划清边界 (Scope boundary)
V1 版本只收 `kind="consolidation"` 的信封 (探针和进化任务的信封等下一波再上，反正数据通道都一样)。V1 **绝不** 搞什么自动下发信任公钥、自动轮换秘钥那一套花活；`trusted_keys` 就是一个人工手填的弱智 JSON。V1 **绝不** 重新去跑大模型的接口 —— 重放机只负责极其残暴的确定性重演 (对着那些已经烂在存储里的印记，拿规划器死板的刀工再切一遍)。没有常驻后台进程，没有定时任务，绝不碰网线。

### 风险评估 (Risks)
- **非确定性毒药泄露**: 只要哪天 `_payload_text` 这个函数改了一下去空格的逻辑，以前所有的信封在验尸的时候全部会炸。**护城河**: 必须在信封里加一个死死的 `text_norm_version` 版本号。
- **私钥大放血**: 小白用户直接把私钥跟 `bene.yaml` 一起 `git commit`。**护城河**: `bene replay keygen` 造出来的文件权限必须被锁死在 `0600`，绝对不准塞进代码库的目录下；`bene setup` 必须顺手把它填进 `.gitignore` 里。
- **沙盒成本失控**: 为了 `verify` 去每次生造一个带有 `ensure_v2` 的临时 sqlite 库，热启动大概要 50ms。以后清单长了肯定扛不住。**护城河**: 默认一律开纯内存 `:memory:` 库，只有用户拿枪指着你要 `--into` 时才落地写盘。
- **跨兵种水土不服**: 对面 `adx-cli-7` 团队可能指望着另一种形状的信封 (比如他们那破 BadgeAuthority 的规矩)。**护城河**: 先拿 V1 把核心干好。反正信封里有 `version` 字段，大不了让他们自己写个 200 行以内的适配器 PR 接上来。

## 2. MemGAS — 看熵下菜碟的多粒度路由 (MemGAS entropy-routed multi-granularity retrieval)

**状态 (Status)**: 画了饼没做 (planned)。目前用的是一套暴力的确定性"脸熟"打分法 (熟悉度启发式)，虽然文档写得挺全，也开了口子，但这还不是 MemGAS。目前这套代码就在 `bene/kernel/memory/retrieval.py` 里的 `AdaptiveRetriever`：要么走快车道，要么走慢车道；慢车道就去平铺的 FTS 全文索引里硬捞，只在联想扩展时才费点心思，它根本不管阶梯里的印记是不是一堆噪音，统统吃进去。
**代码量估算**: 约 320 行
**对抗裁决**: ✅ 无懈可击 (clean) (四个维度全部过关)

### 现场还原 (Scene)
特种兵 `dirty-3` 撞到了一个它祖宗十八代都没见过的测试死锁 —— 于是大喊一句：`bene retrieve "intermittent 429 on engram FTS reindex"`。
按现在的弱智跑法，系统直接给它塞了一大堆毫不相干的 tier-0 级流水账；而上个星期它大哥费劲心血提炼出来的 tier-2 级绝世心法 "FTS rebuild needs WAL checkpoint" 却被淹没在里面。
如果用了看熵下菜碟的路由 (Entropy Routing)，系统会发现 tier-0 那边得分全都差不多 (高熵，全是噪音)，而 tier-2 那边有一个孤零零的高分 (低熵，这是真金)；于是路由直接把 tier-2 抬在头上送给特种兵，省了它在垃圾堆里翻找的半个小时。

### 痛点 (Problem)
今天，所有的查找请求全往那张拍扁了的 `engram_fts` 表里撞，不管特种兵是想要点新鲜细节 (tier-0)、还是要一本内功心法 (tier-2)、甚至是想要变异基因 (tier-4)。这导致简单的破事要交过路费去算联想，而人命关天的破局心法却根本没人看。MemGAS 根据每层的**混乱程度 (entropy)** 来指路：如果这一层有个尖峰 (大家都觉得这篇最准)，那就把这层权重拉满；如果这一层全是均摊的烂泥 (极度混乱，没有突出者)，那就把它的权重压到底，去别处看看。

### 兵器长什么样 (Proposed API)

```python
# bene/kernel/memory/retrieval.py — 只许加代码，不准拆老家。
from dataclasses import dataclass

@dataclass
class TierProbe:
    tier: int                 # 第几层阶梯 (0..4)
    hits: list[Engram]
    entropy: float            # H(softmax(-bm25_scores)) 算出来的熵，被卡死在 [0,1] 之间
    weight: float             # 用 softmin 压出来的权重，所有层的权重加起来必须 == 1
    top_score: float

@dataclass
class MemGASResult(RetrievalResult):
    tier_probes: list[TierProbe]
    routed_tiers: list[int]   # 最后真被端上桌子的层 (权重 >= cutoff 的)

class MemGASRouter(AdaptiveRetriever):
    """看熵下菜碟的多粒度路由。它必须是原来那套系统 (AdaptiveRetriever) 的亲儿子：
    先把该翻的层选出来，然后才把那些命中数据送进原来的 快/慢 车道里。
    向后兼容铁律 —— 如果最后只剩下一层数据能用，它就老老实实退化成原来的傻瓜模式。"""
    def __init__(
        self,
        store, granules=None, *,
        tiers: tuple[int, ...] = (0, 2, 3, 4),    # 默认跳过那种看戏用的 tier 1 (episodic)
        per_tier_k: int = 8,
        weight_cutoff: float = 0.15,
        fast_threshold: float = 0.6,
    ): ...
    def probe_tier(self, query: str, tier: int, k: int) -> TierProbe: ...
    def route(self, probes: list[TierProbe]) -> list[TierProbe]: ...   # softmin(entropy) 的刑具
    def query(self, agent_id: str, text: str, *, k: int = 8) -> MemGASResult: ...
```

CLI 长这样 (在原来基础上的微雕)：

```bash
bene retrieve "intermittent 429 on engram FTS reindex" \
  --agent dirty-3 --k 8 --tiers 0,2,3,4 --json
# 必须吐出这种东西：{"path":"slow","tier_probes":[{"tier":0,"entropy":0.92,"weight":0.05,...}, ...], "routed_tiers":[2,4], "hits":[...]}
```

`bene.yaml` 开关配制：

```yaml
kernel:
  memgas:
    enabled: true
    tiers: [0, 2, 3, 4]
    per_tier_k: 8
    weight_cutoff: 0.15
    temperature: 1.0   # 给 tier 算权重时用的 softmin 温度
```

默认必须锁死在 `disabled`。只要不开口子，系统跑的还是那套弱智 `AdaptiveRetriever`，直到探针用它的人头盖戳 (ACCEPT)。

### 数据长什么样 (Data structures)
**一行新 SQLite 表都不准建。** **一种新印记都不准生造。**
所有数据都往印记现有的 `metadata` JSON 里塞：加了 `tier_probes`, `routed_tiers`, `temperature` 这三个新键。这算是在老规矩上做加法 —— 老花眼的旧系统看见这些键也权当没看见；只要保证**每一次搜寻必留一条可查的记录**这条祖宗规矩没被破就行。
新增一套配置 `kernel.memgas`，交给 `runner_kwargs_from_config` 解析；读不到就当它死了 (disabled)。

### 怎么落地 (Implementation sketch)
**责任人**: `bene/kernel/memory/retrieval.py` (在现成的 86 行代码下面接管，大概加 140 行) + 在 `bene/kernel/memory/__init__.py` 暴露接口 + `bene/config.py` 里解析 `kernel.memgas.*` + 在 `bene/cli/main.py` 里新开个 `retrieve` 命令 (大约 40 行)。

**每次问答的流水线 (Dataflow)**:
1. 挨个把配置的 `tiers` (0,2,3,4) 扒一遍：调底层的 `store.search(safe_query, tier=t, limit=per_tier_k)` —— 这个底层方法早就做好了，带上 `tier=` 直接用。
2. 捏出每层的熵：`entropy = -sum(p_i*log(p_i)) / log(k)`，这边的概率 `p_i = softmax(-bm25_score_i / temperature)`。(因为 BM25 分数是越小越牛逼，所以得加个负号倒过来)。如果哪一层扒出来是个空壳 ⇒ 熵直接拉满 1.0，权重按死在 0。
3. 过软骨头机器：`weights = softmin(entropies, temperature)`；凡是权重没过线 (`weight < weight_cutoff`) 的层，直接扔进垃圾桶 (`routed_tiers` 剔除)。
4. 把活下来的残兵败将拼在一起：按 `engram_id` 把重复的筛掉，按 `weight_t * (1/rank)` 重新洗牌；最后只留下头部的 `k` 个。
5. 去查脸熟度：拿着之前算 Jaccard token 重合度的老办法跑一遍 —— 这玩意被当成最终判决的守门员，只有当脸熟度不够 (< threshold) 的时候，才会被踢进联想扩展池里去。
6. 最后留案底：造一个 `trace` 印记，里面的元数据必须交代得清清楚楚：`metadata={"retrieval_path", "familiarity", "tier_probes":[{tier,entropy,weight,top_score}...], "routed_tiers", "hits": n}` —— 这一步绝不能省，必须留下铁证。

**跟 5 大台柱子的咬合**:
- **印记阶梯**: 横着在 0-4 层阶梯里吸血；全程复用 `EngramStore.search(tier=...)`；**零新表、零新印记类型**。

- **兵权与信用**: 只要是个能读档案的 L1 新兵蛋子就能开 MemGAS 路由；没碰 L4 人类授权的禁区。
- **进化管家**: 那个 `temperature`, `weight_cutoff`, `tiers` 在探针盖下 ACCEPT 的戳之后，全变成了 `tier=4 kind="strategic"` 的战略基因 —— 要上位就得通过 `evolve.promote` 走死门禁的流程。

### 现场还原 (Scene)

运维人员需要这个叫 `bene skills plasticity scan` 的黑洞清理机，能自己跑出来把 `#17` 强行挂上 `demoted` (贬职) 的破牌子，把以前的旧线 `supersedes` 扯到 V2 版本的新技能上，并让所有的查询在翻牌子的时候像躲瘟神一样跳过它 —— **而且，所有的原始打分算术题，一行都不用改，旧账绝不擦**。

### 痛点 (Problem)
一个早就坏掉但一个月前打过四场漂亮胜仗的过气网红技能，依然能霸占排行榜。这是因为那个所谓的衰减期只调整了**搜索的排名**，它从头到尾都没敢把废柴从列表里拿掉或者记录一下它被哪个新版本碾压了。特种兵跟没头苍蝇一样挑着坏的技能去送死，烧钱烧 Token 留下一地垃圾，而且运维查账的时候根本查不到 "到底是谁把这个老古董给撤下来的？" 这种审计留痕。

### 兵器长什么样 (Proposed API)

```python
# bene/kernel/memory/plasticity.py  (这是这套规矩的专属山头)

from dataclasses import dataclass
from bene.kernel.eval.probe import Probe  # 这把刀也是被 sha256 锁死的

@dataclass(frozen=True)
class PlasticityPolicy:
    min_uses: int = 8                 # 次数没用够这把数之前，谁也不准判它死刑 (给新兵活路)
    recent_window: int = 10           # 只扫最近的这几条尾巴账
    failure_threshold: float = 0.25   # 只要最近这十场均分烂过这条死线，直接杀头
    half_life_days: float = 14.0      # 跟 skills.py 里的 DEFAULT_HALF_LIFE_DAYS 对齐
    retire_after_days: float = 90.0   # 带着贬职牌子闲置过这个天数 → 彻底退休装盒
    require_probe_accept: bool = True # 要想走这条剥夺特权的路，必须带着探针过死门禁

class PlasticityScanner:
    def __init__(self, conn, store, probe: Probe | None = None): ...

    def scan(self, *, dry_run: bool = False,
             now: datetime | None = None) -> "PlasticityRun":
        """把每个满足出场次数的老兵查一遍；吐出 贬职(Demote) / 退居二线(Retire) / 暂时放过(Hold) 的判决书；
        把罪证压成一条印记，再加上 skill_lifecycle 表里留档 (只有当过了探针而且不是在演习 dry_run 时才落盘)。"""

    def supersede(self, old_skill_id: int, new_skill_id: int,
                  *, verdict_engram_id: str) -> str:
        """把这根 'supersedes' 的新老交替红线牵在 engram_links 里；你手里必须拿着那张 ACCEPT
        的裁决书印记 (跟 evolve.promote 一模一样的死规矩)。返回那条红线的 ID。敢走后门抛出 PromotionBlocked。"""

# 命令行武器库 (全塞在 skills 原有的摊位里，不新开店面)
$ bene skills plasticity scan --dry-run [--policy plasticity.yaml]
$ bene skills plasticity scan                # 让探针盖章杀人
$ bene skills plasticity supersede 17 42 --verdict <engram_id>
$ bene skills ls --include-demoted           # 这玩意默认是遮掉废柴的，带了这个才露底
$ bene skills lifecycle 17                   # 把这个兵的上下马履历跟罪证一条条拉出来
```

唯一的一刀切在 `SkillStore.search(rank="weighted")` 上：悄悄在查询最后补个 `WHERE NOT EXISTS (SELECT 1 FROM skill_lifecycle l WHERE l.skill_id = s.skill_id AND l.status IN ('demoted','retired'))`，除非有人掏出 `include_demoted=False` 这个后门钥匙。算分的公式一根毛都没碰。

### 数据长什么样 (Data structures)
**开一张叫 `skill_lifecycle` 的新表** (这是叠在上面的，写在 `bene/schema.py` 里的 MIGRATION_V5_SQL 里，绝不准动 `agent_skills` 这个底座)：
```sql
CREATE TABLE IF NOT EXISTS skill_lifecycle (
    lifecycle_id   INTEGER PRIMARY KEY AUTOINCREMENT,
    skill_id       INTEGER NOT NULL REFERENCES agent_skills(skill_id),
    status         TEXT NOT NULL CHECK (status IN
                   ('active','demoted','retired','restored')),
    reason         TEXT NOT NULL,            -- 填判决书，比如 "最近 10 次的均分只有 0.08"
    decided_by     TEXT NOT NULL,            -- 谁杀的："policy:PlasticityScanner" 或者是人 "human:<name>"
    verdict_engram TEXT,                     -- 装着那张 ACCEPT 判决书的印记 ID (如果人类拿 L4 特权硬来，这个可以空着)
    superseded_by  INTEGER REFERENCES agent_skills(skill_id),
    decided_at     TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%f','now'))
);
CREATE INDEX idx_skill_lifecycle_skill  ON skill_lifecycle(skill_id, decided_at DESC);
CREATE INDEX idx_skill_lifecycle_status ON skill_lifecycle(status, decided_at DESC);
```
用这张表里每个技能的最新行当做它现在的死活状态；**绝不准 UPDATE 原来的旧账** —— 这是必须死守的追加制记账审计铁律。

**印记白嫖老住客** (不建新种类)：所有的裁决死刑状全以 `kind='intervention'` (干涉) 印记入住，里面带着 `metadata={"plasticity": True, "skill_id": N, "from": "active", "to": "demoted", "policy_hash": "..."}`。至于技能之间老带新的顶包操作，全用上了原本阶梯里在 V2 版本就造好的 `engram_links.link_type='supersedes'`。

**YAML 配制开个新洞**：
```yaml
plasticity:
  enabled: true
  min_uses: 8
  recent_window: 10
  failure_threshold: 0.25
  retire_after_days: 90
```

### 怎么落地 (Implementation sketch)
**责任人:** `bene/kernel/memory/plasticity.py` —— 一块全新的工地，安插在跟 `lifecycle.py` (专门管把短期记忆压成长期记忆的打包站) 并排的位置，这俩干的都是 "对着睡着的印记干清扫保养的活"。

**每次扫街的管线 (Dataflow):**
1. 扫地大妈 `PlasticityScanner.scan()` 先去 `agent_skills` 连着 `skill_uses` 兜一圈底 (不准全盘胡扫，只顺着现成的索引 `idx_skill_uses_skill` 揪出每个技能的尾巴)。
2. 把那些熬过了新手保护期 (`use_count >= min_uses`) 的老兵拉出来，算一算最近 `recent_window` 场里的 `mean_quality`。要是连个质量分 (quality) 都没有，就用 `success` 当备胎硬算 (这套转换的把戏 `_quality_signal_map` 早就在玩了)。
3. 切西瓜：如果分烂透了 (`<= failure_threshold`) ⇒ 拉出去斩了 (`Demote`)；如果早就被贬职而且闲了一万年 (`≥ retire_after_days`) ⇒ 收进盒子里 (`Retire`)；其余全留着发光发热 (`Hold`)。
4. **判死刑的门禁**：每一个被挑出来的倒霉蛋，都得送上一台带 sha256 锁死的 探针断头台 (`Probe`) 上去跑一遍。探针的门槛是这么写的：`"current_window_mean_quality > demoted_window_mean_quality"`。这台机器的基准是那个技能在变烂之前的三十天战绩；而现在测试的对象就是它最近的这笔烂账。只有当 `mean(subject) <= failure_threshold` 而且 `mean(subject) < mean(baseline)` 的时候，门槛直接卡死。如果此时被探针砍死 (REJECT)，说明 "这技能真的在退化，这个衰败的断言是可被证明的" —— 所以死刑被执行。**(这里的逻辑是为了强行倒贴死门禁的怪异规则：ACCEPT = 没出岔子，所以我们只有在 REJECT 的时候才动手杀人。)**
5. 写记录：搞一条干涉印记 (带着政策哈希和判书)，在 `skill_lifecycle` 里留尸体，然后再顺着 `gated_by` 拉根线牵着那份裁决书印记 —— 这一手全是抄的 `evolve.promote()` 晋升的剧本，分毫不差。
6. `SkillStore.search` 在检索的时候就多拼一段 `LEFT JOIN` 把最新的尸检报告拉上，如果状态是不详直接过滤。

**跟 5 大台柱子的咬合:**
全盘通吃。全是原班人马的 `intervention` 印记；用的又是死板的死门禁 `Probe.run()`，没有偷渡小道；用的又是 L2 自动裁决和 L4 人类许可的分水岭规则。
**打死不准碰活人的账本:** 这套东西敢去改 `agent_skills.use_count` 或者删源数据里的半个字，直接剁手。复活技能 (`restored`) 只是加一行带有原谅状态的新记录。

### 划清边界 (Scope boundary)
这个图纸**绝不**去帮你自动立什么接班人 —— 换人顶包的 `supersede()` 你必须拿着带章的 ACCEPT 判决书，还得有 L4 的特权 (除非配置放开) 才能划上那条线；**绝不**私自动原表任何东西；**绝不**塞后台耗子进程；全靠打手 CLI 从外部敲。不搞分门派的精细化打击阈值 (现在只搞一刀切)，不搞自动原地复活 (只能靠人 `bene skills plasticity restore` 救)。跟 meta-harness 繁殖大圈的交配，那是另一个功能的事。

### 风险评估 (Risks)
- **刷差评陷害对手**: 如果一个流氓特种兵为了上位，疯狂拿着竞争对手的技能疯狂塞 `record_outcome(success=False, quality=0)` 的黑账，那技能瞬间死路一条。**护城河**: 查账的时候拉一下报案人的花名册 `agent_id`，只有这笔黑账是超过 2 个以上的不同人报上来的，才算数，把防刷榜的事写在 `reason` 里。
- **两头通吃剥削两遍**: 如果技能原来那个搜索里的倒计时半衰期也是 14 天，这边扫描也拿这个去惩罚，等于是挨了两次罚。**护城河**: 我们扫描的时候，切的是固定最后几场比赛 (row count)，不去死抠它在时间线上有多旧，让这两个车道彻底分开跑。
- **配置动一下门槛碎一地**: 如果这套裁决的 policy 改了个数字 (0.25 -> 0.3)，探针的规格就跟着变了，接下来就一定是触发 sha256 的报错自动 REJECT。这就只能认栽重跑一次册登记。必须把这种流血代价印在 `--help` 手册上。

**裁判点评**:
痛点一针见血：因为 `GAP-AUDIT` 的 175 行就差指着鼻子骂 "开始摆烂的技能永远不可能被扔进垃圾桶" 了；设计里的那个旧账机制和防老表不碰面也绝对对齐，咬合极为精准 (没破坏死门禁，没有破坏来源铁律)。而且绝没造轮子：那套权重分这期没碰，搞出来的就是清扫审计。

但是，**在可证伪性的这个测试环节 (eval_falsifiable) 脑部打结了 (BROKEN)**！
你在落地设计第 4 步里为了强行装杯，写了个诡异的倒转逻辑 ("ACCEPT = no degradation observable, demote on REJECT")。但整个内核其余部分 —— `evolve.promote`, `distill` 等等全特么是 "ACCEPT 是绿灯放行，REJECT 是斩首拦截"！你在这玩逻辑倒贴，不仅把原本两个不同的探针 (一个用来拦每次杀头的个案，一个用来测整个功能有没有出 bug 的宏观探针) 给杂交在了一起；你在事后留账本的时候，一条挂着 `gated_by` 指向一个被 REJECT 掉的废纸，谁看了不觉得是 "这个贬职操作被驳回了"？而且更恶心的是，如果你这套探针是个废品连自己都不认识 (VOID)，它就会无声无息地让你这套剥夺技能永远成死水，连个警报都不响！
边界也越界了点：`skill_lifecycle` 建在新系统的内核之外竟然塞进了前朝遗老的 `bene/schema.py` MIGRATION_V5 里面；护城河说要看 `agent_id` 的花名册防刷单，结果你的 `Dataflow` 流水线上半个字都没提到这一步。

**修补意见 (Suggested fix)**:
打回重整逻辑。这个砍头探针必须老老实实回到内核通用的 "绿灯放行" 常识里：把探针门限改成 `mean(subject) <= mean(baseline) - epsilon`，如果达到了这个降级标准，就发出一张 ACCEPT 裁决书 (意思是 "老古董被证实真拉胯了，贬职通过")，如果人家没变烂，就发 REJECT (意思是 "他还能打，留着")。并且必须在文档里把对每个技能行刑用的这套探针和上面 P1/P2 用来测系统的测试探针画清界限。把所有越界和小臭虫修干净再拿来。

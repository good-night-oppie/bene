# 预注册评估协议 (Pre-registered Evaluation Protocol) — BENE vs KAOS / 前身 0.1.0 独占基准测试

注册于 2026-06-11，在执行任何基准测试**之前**。这份文件的 sha256 校验和已经绑定在同一个 commit 里；如果事后偷偷跑来改判决标准，那就等同于**协议违规**（任何基于被篡改过的协议跑出来的结果，一律判作**无效 (VOID)**）。

## 铁律 (不可谈判)

1. **绝对禁止“调参刷榜” (No retune-and-rerun)。** BENE 这边的参数死死锁死在当前的 HEAD（3acb93d 这个血脉分支上）。只要某一行输了，那就是输了。
2. **同源数据，同等考卷。** BENE 对应机制在吃竞品那套数据集 / 查询词时，必须**一字不差**地硬啃（只准平移，不准搞什么重新组织或者是降维打击）。
3. **把“拒绝”也算作一种成功。** 只要诚实披露，一行“失败 (LOSS)”的数据，同样能算是一次成功的审计。如果有谁敢在哪个格子里加了层糖衣粉饰太平，那这份报告就直接报废了。
4. **N/A 就意味着这块能力版图你输了。** 如果某一项同行基准测试要测的机制，BENE 根本就没做出来，那这一行就得老老实实打上“N/A — 能力鸿沟 (计划中)”，而且这个记录会在最终结算时，结结实实地成为扣掉“全面超越”这句宣言底气的一笔。
5. **基线标定：** 竞品提交的 `results.json` 必须被引用，且只要挂钟时间允许，必须**重新测跑**；要是重跑的数据和他们挂出来的结果有超过 10% 的出入，那就必须举牌标明，并以重跑数据为准。

| 行 | 测试任务 | BENE 受检机制 | 门禁 (BENE 通关的唯一条件) |

### C 组 — BENE 独有战法 (既然来了，竞品也得挨下 BENE 设下的擂台)

对于下面每一项：BENE 自家的测试必须在 HEAD 上满血过关，**并且**竞品必须露出在指令级别上根本接不住这招的破绽（不管是 grep 找不着东西，还是压根没机制），这里的查证标准和 GAP-AUDIT 那套一模一样。

| 行 | 能力域 | 竞品待查验的死穴 |
|---|---|---|
| C1 | 带处决门禁的晋升机制 (不拿到 ACCEPT，晋升必然被 PromotionBlocked 截停) | KAOS：确实有探针，但没跟晋升卡死；前身 0.1.0：压根连探针都没有 |

| C3 | 铁腕把控的自治阶梯 (L1 休想干 L3 的活，并且这笔越权账会被记下来) | KAOS：口惠而实不至的纸上文档；前身 0.1.0：二极管管死，要么全跑要么就杀 |

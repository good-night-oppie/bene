## Overview

These eight features collectively turn the BENE-v0.2 kernel's substrate from VERIFIED-but-passive (engrams + probes + trust + ContextOS shipped) into actively-wired (scheduler that runs them, replay that signs them, plasticity that prunes failing skills, entropy routing that picks granularity, in-episode mutation that adapts mid-run). Five fit the existing kernel boundaries cleanly; three needed substantive revision per the adversarial pass — see the **Verdict** subsections.

Hard contracts retained across all designs:

- L4 grants require `granted_by="human:<name>"` — no design introduces auto-L4
- Probes stay sha256-locked; any spec mutation produces an automatic REJECT
- `promote()` is the only path through the kill gate; no back-door integrations
- Provenance mandatory; promotions add new rows, never mutate sources

## 1. bene replay — signed deterministic replay CLI

**Status**: CLAIMS-AUDIT line 52: "Deterministic replay surfaces — partial. Journal + checkpoints remain the substrate; scheduled consolidation now emits deterministic replay manifests, but end-to-end replay tooling/CLI is not shipped." Status: partial. Substrate (events journal, checkpoints, `ConsolidationPlan.replay_manifest`, `experiment_runs` row) is VERIFIED (tests/kernel/test_memory_os.py::test_scheduled_consolidation_plans_replay_manifest_deterministically). What is missing is (a) a CLI that re-executes a manifest into a sandboxed db and asserts byte-identical engrams, (b) an ed25519-signed export envelope, (c) a `cite_as` BibTeX shim so a replay artifact is referenceable in adx-cli-7's adversarial 11d slot.  
**LOC estimate**: ~420 lines  
**Verdict**: ✅ clean (adversarial pass clean on all 4 axes)

### Scene
04:32 你正在跟 adx-cli-7 pane 喊话，他们 11d.4 PR 要把 BENE 的 consolidation 当 adversarial witness 引用——他们打开 `experiment_runs` 看到一串 ULID 和一个 metrics blob，问 "这玩意我怎么 re-run？怎么证明你没改？" 你敲 `bene replay export <run_id> --sign` 丢过去一个 `.bene-replay` 文件，对面 `bene replay verify <file>` 自己跑一遍，digest 对上，BibTeX 字段直接 paste 进他们的 paper draft。

### Problem
A `experiment_runs` row today says "we ran a consolidation" but a downstream consumer (adx-cli-7, peer reviewer, future you) cannot independently re-derive the resulting engrams from the source granules and tamper-detect the manifest. Without a `bene replay` verb + signed envelope, every cross-lineage citation is a trust-me link.

### Proposed API

```python
# bene/kernel/replay/manifest.py
@dataclass(frozen=True)
class ReplayEnvelope:
    version: int                 # 1
    run_id: str
    kind: str                    # "consolidation" today; "probe"/"evolution" later
    manifest: dict               # exact ConsolidationPlan.replay_manifest() output
    source_payload_hashes: dict[str, str]  # engram_id -> sha256(payload bytes)
    bene_version: str            # pyproject __version__
    schema_version: int          # kernel_schema_version
    created_at: str
    signature: str | None        # base64(ed25519); None if --no-sign
    public_key: str | None       # base64; None if --no-sign

class ReplayExporter:
    def __init__(self, store: EngramStore): ...
    def export(self, run_id: str, *, sign_key: Ed25519PrivateKey | None) -> ReplayEnvelope: ...

class ReplayVerifier:
    def verify(self, env: ReplayEnvelope, *, into_db: Path) -> ReplayResult:
        """Re-plan against a sandboxed db hydrated from source_payload_hashes
        and assert: (1) signature valid, (2) re-planned manifest == env.manifest,
        (3) source content hashes match, (4) probe lock_sha256 unchanged if kind=probe."""

@dataclass(frozen=True)
class ReplayResult:
    ok: bool
    reasons: tuple[str, ...]     # empty iff ok
    replayed_manifest: dict
    cite_as: str                 # BibTeX @misc{bene:<run_id>, ...}
```

```bash
# CLI (additive — no new top-level subsystem)
bene replay export <run_id> --out run.bene-replay [--sign] [--key-file ed25519.key]
bene replay verify run.bene-replay [--into /tmp/replay.db] [--json]
bene replay cite   <run_id> [--style bibtex|json]    # offline cite_as without full export
bene replay ls                                       # exportable run_ids (= experiment_runs)
```

A bare `bene replay <run_id>` (no subcommand) is a convenience alias for `verify` against the live db (re-derives in-memory, asserts equality, exits non-zero on mismatch).

### Data structures
No new tables required for the v1 cut. Two tiny additions:

1. New engram kind tag (no schema change — `metadata` json key): when `bene replay export` runs, it writes a `kind='report'` engram titled "replay-envelope:<run_id>" with the canonical envelope JSON as inline_body and `provenance={"system":"bene.kernel.replay","source_run":run_id}`. This makes the envelope itself a first-class, link-able engram (so a downstream `engram_links` row can express "this paper cites that replay").

2. Optional config block in `bene.yaml`:
   ```yaml
   kernel:
     replay:
       sign: true
       key_file: ~/.config/bene/replay_ed25519.key   # generated by `bene replay keygen`
       trusted_keys: ~/.config/bene/replay_keys.json
   ```
   Loaded through the existing `bene/config.py` path; absent block = unsigned exports allowed, verify still works on unsigned envelopes but ReplayResult flags `signature="absent"`.

Defer a dedicated `replay_runs` table until a second kind (probe/evolution) lands — current `experiment_runs` is sufficient.

### Implementation sketch
Owner module: new `bene/kernel/replay/` (manifest.py, exporter.py, verifier.py, cite.py, keys.py). Lives **inside the engram-ladder subsystem** — it is a read-only projection of `experiment_runs` + the engrams they cite, not a new subsystem.

Dataflow:
1. `export(run_id)` reads `experiment_runs` row, parses `metrics` json (already deterministic by `json.dumps(..., sort_keys=True)` at line 352 of lifecycle.py), then for every `source_id` referenced in `manifest["batches"][*]["source_ids"]` it reads the engram payload via `EngramStore.payload()` and records `sha256(payload)`. Signs the canonical JSON (sorted keys, no whitespace) with ed25519 using `cryptography.hazmat.primitives.asymmetric.ed25519` (already in uv.lock, no new dep).
2. `verify(env, into_db)` opens a temp sqlite, runs `ensure_v2`, inserts shadow engrams whose payloads exactly match `source_payload_hashes`, instantiates `ScheduledConsolidator`, calls `.plan(policy=ConsolidationPolicy(**env.manifest["policy"]), force=True)`, and asserts `plan.replay_manifest() == env.manifest`. Signature verified via stored `public_key` (warn if key not in a configured trust list at `~/.config/bene/replay_keys.json`; verify cryptographically regardless).
3. `cite_as` is rendered from envelope: `@misc{bene:<run_id>, title={BENE replay <kind> <short>}, note={sha256:<digest_of_canonical_envelope>}, year={YYYY}, url={file://...}}`.
4. CLI lives under `bene/cli/main.py` as a `@cli.group("replay")` with 4 subcommands; reuses `_kernel_stores(db)` and the `--json` ctx wiring (line 37).

Integration with the 5 subsystems:
- **Engram ladder**: read-only consumer of engrams + `experiment_runs`; never mutates source rows (provenance contract preserved).
- **Probe + kill gate**: when `kind="probe"`, envelope additionally pins `probe_registry.lock_sha256`; if the live probe row's sha differs at verify time, ReplayResult.ok=False with reason "probe-spec-mutated" — this is the existing automatic-REJECT contract surfaced as a replay verdict, not a new gate.
- **Context OS / autonomy / breeding**: untouched.

Probe A — `replay.roundtrip.consolidation` (ACCEPT iff verify equals export):

### Scope boundary
v1 covers `kind="consolidation"` envelopes only — probe and evolution envelopes ship in a follow-up (the data path is identical; the verifier shim per-kind is the only delta). v1 does NOT ship a trust-on-first-use UI or a key-rotation policy; trusted_keys is a flat json list edited by hand. v1 does NOT re-execute the LLM-side of any flow — replay is purely deterministic over already-recorded engram payloads + the planner's deterministic batching. No daemon, no cron, no network.

### Risks
- Non-determinism leak: if `_payload_text` (lifecycle.py:42) ever changes its whitespace normalization, every previously-exported envelope fails verify — mitigation: pin a `text_norm_version` field in the envelope and version-gate at verify.
- Signature-key UX footgun: users `git commit` their private key alongside `bene.yaml` — mitigation: `bene replay keygen` writes 0600 to `~/.config/bene/`, never inside the repo, and `bene setup` adds the path to `.gitignore` advisory.
- Verifier sandbox cost: rebuilding a temp sqlite + ensure_v2 per `replay verify` call is ~50ms on warm fs but could regress when batch_size grows — mitigation: in-memory `:memory:` db by default, on-disk only when `--into` is passed.
- Cross-lineage adoption mismatch: adx-cli-7's 11d.4 may expect a different envelope shape (their BadgeAuthority key format) — mitigation: the envelope `version` field plus their stated cadence (2-3 day tiny PRs) means we ship v1 narrowly, let them open a thin-CLI PR for the integration adapter.

## 2. MemGAS entropy-routed multi-granularity retrieval

**Status**: CLAIMS-AUDIT.md line 39: "MemGAS entropy-routed multi-granularity retrieval | planned | UNVERIFIED | deterministic familiarity heuristic shipped instead (documented, pluggable)". Status: planned. Today `AdaptiveRetriever` (bene/kernel/memory/retrieval.py) makes a binary fast/slow choice on token-Jaccard familiarity, searches one flat FTS index, and only expands associatively on the slow path — it does not weight ladder tiers by uncertainty.  
**LOC estimate**: ~320 lines  
**Verdict**: ✅ clean (adversarial pass clean on all 4 axes)

### Scene
Agent dirty-3 hits a flaky test it has never seen before — fires `bene retrieve "intermittent 429 on engram FTS reindex"`; today it gets noisy tier-0 turns from three unrelated agents and misses the tier-2 semantic engram "FTS rebuild needs WAL checkpoint" written last sprint. With entropy routing the tier-0 distribution scores high entropy (flat, no signal), tier-2 scores low entropy (one strong hit), router pulls from tier-2 first and the agent skips a half-hour wild goose chase.

### Problem
Today every retrieval hits the same flat engram_fts index regardless of whether the agent needs a raw turn, a consolidated semantic, or a strategic gene — so easy queries pay associative-expansion cost and hard queries miss the higher tiers entirely. MemGAS routes by per-tier entropy: when one tier's hit distribution is sharply peaked, pull from it; when it is flat (uncertain), down-weight it and look elsewhere.

### Proposed API

```python
# bene/kernel/memory/retrieval.py — extend, do not replace.
from dataclasses import dataclass

@dataclass
class TierProbe:
    tier: int                 # 0..4
    hits: list[Engram]
    entropy: float            # H(softmax(-bm25_scores)), normalized [0,1]
    weight: float             # softmin over entropies, sum==1
    top_score: float

@dataclass
class MemGASResult(RetrievalResult):
    tier_probes: list[TierProbe]
    routed_tiers: list[int]   # tiers actually merged into hits (weight >= cutoff)

class MemGASRouter(AdaptiveRetriever):
    """Multi-granularity entropy router. Subclass: keeps fast/slow as a
    final stage AFTER tier selection. Backwards-compatible — falls back
    to single-tier behavior when only one tier returns hits."""
    def __init__(
        self,
        store, granules=None, *,
        tiers: tuple[int, ...] = (0, 2, 3, 4),    # skip tier 1 episodic by default
        per_tier_k: int = 8,
        weight_cutoff: float = 0.15,
        fast_threshold: float = 0.6,
    ): ...
    def probe_tier(self, query: str, tier: int, k: int) -> TierProbe: ...
    def route(self, probes: list[TierProbe]) -> list[TierProbe]: ...   # softmin(entropy)
    def query(self, agent_id: str, text: str, *, k: int = 8) -> MemGASResult: ...
```

CLI surface (new subcommand, additive):

```bash
bene retrieve "intermittent 429 on engram FTS reindex" \
  --agent dirty-3 --k 8 --tiers 0,2,3,4 --json
# stdout: {"path":"slow","tier_probes":[{"tier":0,"entropy":0.92,"weight":0.05,...}, ...],
#          "routed_tiers":[2,4], "hits":[...]}
```

`bene.yaml` opt-in (kernel section already exists):

```yaml
kernel:
  memgas:
    enabled: true
    tiers: [0, 2, 3, 4]
    per_tier_k: 8
    weight_cutoff: 0.15
    temperature: 1.0   # softmin temp for tier weights
```

Default = disabled; `AdaptiveRetriever` remains the live path until probe ACCEPTs.

### Data structures
No new SQLite tables. No new engram kinds. Engram `metadata` JSON gains optional keys `tier_probes`, `routed_tiers`, `temperature` (additive — existing readers ignore unknown keys; CLAIMS-AUDIT row 38 "auditable path metadata" preserved). New config section `kernel.memgas` parsed by `runner_kwargs_from_config`; default missing == disabled. Strategic-gene payload (for breeding) is JSON `{"router":"memgas","tiers":[...],"temperature":..,"weight_cutoff":..}` stored at tier=4 kind="strategic" — uses existing genome wiring.

### Implementation sketch
Owner: `bene/kernel/memory/retrieval.py` (extends current 86-LOC module; new ~140 LOC) + `bene/kernel/memory/__init__.py` re-export + `bene/config.py` (`kernel.memgas.*` parsing) + new `bene/cli/main.py` `retrieve` command (~40 LOC).

Dataflow per query:
1. For each configured tier t in `tiers`, run `store.search(safe_query, tier=t, limit=per_tier_k)` — this already exists, just narrowed by `tier=`.
2. Compute per-tier `entropy = -sum(p_i*log(p_i)) / log(k)` where `p_i = softmax(-bm25_score_i / temperature)` (BM25 ranks are smaller=better, so negate). Empty tier ⇒ entropy=1.0, weight pinned to 0.
3. `weights = softmin(entropies, temperature)`; tiers with `weight < weight_cutoff` are dropped (`routed_tiers`).
4. Merge surviving tiers' hits, deduped by engram_id, sorted by `weight_t * (1/rank)`; truncate to `k`.
5. Compute familiarity (existing token-Jaccard) — kept as the FINAL fast/slow stage on the merged result for associative expansion only when familiarity < threshold.
6. Append one `trace` engram with `metadata={"retrieval_path", "familiarity", "tier_probes":[{tier,entropy,weight,top_score}...], "routed_tiers", "hits": n}` — auditability invariant preserved (every retrieval is one queryable engram).

Subsystem mapping (no new top-level subsystem):
- **Engram ladder**: routes ACROSS existing tiers 0-4; uses `EngramStore.search(tier=...)`; zero new tables, zero new engram kinds.
- **Context OS**: emitted hits feed `ContextOS.assemble` unchanged; `tier_probes` becomes a new signal usable by `select_strategy` (high mean entropy → "compress-then-pack").
- **Eval / kill gate**: ships with a registered probe (below) gating any future default flip.
- **Autonomy / trust**: enabling MemGAS is an L1 capability (read-only retrieval); no L4/human-grant changes.
- **Breeding**: `temperature`, `weight_cutoff`, `tiers` become a strategic gene (kind="strategic", tier=4) once a probe ACCEPTs — promotion goes through `evolve.promote` and the existing kill gate.

### Eval plan
Two probes, both `bene probe register` + `bene probe run` style, both sha256-locked at registration so spec-mutation auto-REJECTs.

Probe 1 — `memgas_routing_accuracy` (planted-ground-truth retrieval):
```bash
bene probe run memgas_routing_accuracy --json
```
Fixture seeds N=200 engrams with known per-tier relevance labels (50 turns/tier-0, 50 semantic/tier-2, etc.); subject = MemGASRouter, baseline = AdaptiveRetriever. Metrics: `recall_at_8`, `wrong_tier_rate`. Gates: `recall_at_8 >= baseline + 0.10` (kill if not) AND `wrong_tier_rate <= baseline - 0.05`. ACCEPT iff both hold; otherwise REJECT.

Probe 2 — `memgas_latency_budget` (no perf regression on the fast path):
```bash
bene probe run memgas_latency_budget --json
```
Replays a recorded query trace (1000 queries) against an isolated DB. Metric: `p95_latency_ms`. Gate: `p95_latency_ms <= baseline * 1.25` (kill if MemGAS is more than 25% slower at p95). ACCEPT iff under the cap.

Both probes admissibility-self-test at registration (baseline-vs-baseline must trip a kill, per existing eval contract); promotion of `kernel.memgas.enabled: true` to repo-default flows through `evolve.promote` and is BLOCKED unless both ACCEPT — no back-door, no auto-flip.

### Scope boundary
Does NOT: replace `AdaptiveRetriever` (subclass only, opt-in); add personalized PageRank propagation from the MemGAS paper (defer — current `granules.neighbors` 1-hop is the substitute, scoped for a follow-up); add an LLM filter step on retrieval candidates (deferred — would need router/tier integration and a cost gate); compute embeddings (BM25-rank-based entropy only — embeddings are a separate roadmap item); modify engram schema; touch L4 grants or autonomy thresholds. Default `kernel.memgas.enabled` ships as `false` until both probes ACCEPT; flipping the default is a separate PR gated on the kill gate.

### Risks
- Entropy-from-BM25-ranks is a proxy, not true posterior uncertainty — a tier with one strong match and one noisy match looks low-entropy and may over-route; mitigation = `weight_cutoff` floor + Probe 1's wrong_tier_rate gate.
- Per-tier search multiplies SQLite work by `len(tiers)` (4x default); on cold cache this can push p95 — Probe 2's 25% latency gate is the kill condition.
- Strategic-gene auto-tuning of `temperature` via breeding could reward-hack Probe 1 by overfitting to the fixture corpus; mitigation = probe fixture is sha256-locked with the gate spec, and promotion requires ACCEPT on a held-out replay set noted in the probe metadata.
- Adding `tier_probes` to engram metadata grows the FTS body for query engrams — bounded by `FTS_BODY_LIMIT=8192` already, but high per-tier-k could approach it; cap `tier_probes` serialization to top-3 hits per tier in metadata.

## 3. Nightly consolidation automation (cron-spawnable CLI + config wiring)

**Status**: CLAIMS-AUDIT.md line 26 ("Scheduled/nightly consolidation automation (SkillClaw-style)") and line 53 (`bene.yaml kernel:` config section): **partial**. `ScheduledConsolidator.plan/run` in `bene/kernel/memory/lifecycle.py` already records deterministic replay manifests + `experiment_runs` rows and is covered by 4 tests in `tests/kernel/test_memory_os.py`; missing pieces are (a) a `bene consolidate` CLI surface, (b) a `kernel.consolidation` block in `runner_kwargs_from_config` / `bene setup`, and (c) a cron-spawnable wrapper that exits 0/1 cleanly for systemd-timers/launchd/cron.  
**LOC estimate**: ~320 lines  
**Verdict**: ✅ clean (adversarial pass clean on all 4 axes)

### Scene
3:07am, the nightly cron on the build box fires `bene consolidate run --policy nightly --json >> /var/log/bene/consolidate.log`; planner sees 47 untouched `tier=0 granule=turn` engrams since last 24h, batches the oldest 8 into one `episode`, writes one `experiment_runs` row with the replay manifest, exits 0. Next morning the operator greps the log for `"due": false` to confirm the interval gate held on weekends.

### Problem
Today the consolidation planner exists but only fires if a human calls it from Python — so turn-tier engrams pile up indefinitely on long-running BENE deployments, and replay manifests never get produced unless someone remembers. Operators need a single shell line they can drop into cron/systemd that is idempotent (skips when not due, records skips as nothing, records runs as one `experiment_runs` row) and exits with a useful status code.

### Proposed API

```bash
# CLI (new sub-group under existing kernel commands)
bene consolidate plan  [--policy nightly] [--agent AID] [--force] [--json]
bene consolidate run   [--policy nightly] [--agent AID] [--force] [--dry-run] [--json]
bene consolidate ls    [--limit 20] [--json]       # last N consolidation experiment_runs rows
bene consolidate show  <run_id> [--json]           # replay manifest from metrics column

# Exit codes (cron contract)
#   0  ran a consolidation OR skip-by-interval (healthy no-op)
#   1  policy validation failed / db error / unknown policy name
#   2  due=False because insufficient-turns (distinguishable for alerting)
```

```yaml
# bene.yaml — new block, additive, gated by kernel.enabled
kernel:
  enabled: true
  consolidation:
    enabled: true                  # default false; off unless opted-in
    policies:
      nightly:
        to_level: episode          # episode | semantic | procedural
        min_turns: 4
        batch_size: 8
        max_batches: 1
        interval_hours: 24.0
        agent_id: null             # null = all agents
        provenance: {scheduler: "cron", system: "bene.kernel.memory.lifecycle"}
      weekly_semantic:
        to_level: semantic
        min_turns: 6
        batch_size: 12
        interval_hours: 168.0
```

## 5. Continuous-quality outcome signal in trust composite

**Status**: CLAIMS-AUDIT row 29 ("Continuous-quality outcome signal — `record_outcome(…, quality=…)` + per-use telemetry"): **VERIFIED** at the skill-store layer (`bene/skills.py` + `tests/test_skills_weighted.py`). The continuous signal is **not yet** consumed by the trust composite — `bene/kernel/trust.py::outcome_reliability` still reads only binary `tool_calls.status` (success/error). This design closes that gap: refine the existing 4th signal, do not add a 5th.  
**LOC estimate**: ~110 lines  
**Verdict**: ⚠️ flagged for revision

### Scene
Mid-sprint, an agent applies the `ensemble_classifier` skill twelve times across a triage batch. Eight runs land at quality≈0.55 (right shape, wrong threshold — recoverable near-misses); four land at quality=0.0 (wrong tool entirely). The author runs `bene trust agent-01` to decide whether to grant L3 on the `classification` domain. Today she sees `outcome_reliability=0.0` (all 12 binary-failed) and denies. With this design she sees `outcome_reliability=0.37` (recency-weighted mean quality), reads the breakdown, and grants L2.5-equivalent probation — the score told her the agent is improving, not catastrophic.

### Problem
A skill that keeps returning near-miss answers (graded 0.4–0.6) is currently indistinguishable in trust scoring from one that keeps returning garbage — both collapse to `status='error'` in `tool_calls`, dragging `outcome_reliability` down identically and making L3 eligibility look the same. We already grade continuously at the skill layer; the trust composite should see that signal instead of throwing it away.

### Proposed API

```python
# bene/kernel/trust.py — refined signal, same weight, same composite formula.
# Reads from two sources, both already extant:
#   1. skill_uses.quality        (continuous, [0,1], per-skill-application telemetry)
#   2. tool_calls.status         (binary, legacy — mapped success=1.0, error=0.0)
# Merged on the agent_id axis, recency-weighted with the existing 20-call half-life.

class TrustLedger:
    def outcome_reliability(self, agent_id: str) -> tuple[float, str]:
        """Recency-weighted continuous outcome score in [0,1].

        Each terminal observation contributes a quality value:
          - skill_uses row with quality IS NOT NULL  -> quality
          - skill_uses row with quality IS NULL      -> success (0/1)
          - tool_calls row (status success/error)    -> 1.0 / 0.0
        All sources are union-sorted by timestamp DESC; exponential rank decay
        with HALF_LIFE_CALLS=20 is unchanged. Empty -> (0.5, "unknown").
        Note format: "recency-weighted over N obs (graded G, binary B)".
        """

# No new CLI surface. Existing `bene trust <agent>` JSON output gains a sub-field:
#   signals.outcome_reliability.note = "recency-weighted over 16 obs (graded 12, binary 4)"
#   signals.outcome_reliability.sources = {"skill_uses": 12, "tool_calls": 4}
# WEIGHTS dict unchanged; composite formula unchanged; L3/L4 thresholds unchanged.
```

CLI smoke (unchanged invocation, richer payload):
```bash
uv run bene --json trust <agent_id> | jq '.signals.outcome_reliability'
```

### Data structures
No new tables. No engram kinds. No `bene.yaml` section. Reuses:
- `skill_uses(skill_id, agent_id, used_at, success, quality, task_hash)` — already shipped (Round-3).
- `tool_calls(agent_id, status, started_at, ...)` — legacy table, already read by current `outcome_reliability`.

One additive index for the union query (kept inside `_SKILL_USES_DDL` so it stays lazy/best-effort):
```sql
CREATE INDEX IF NOT EXISTS idx_skill_uses_agent_time ON skill_uses(agent_id, used_at DESC);
```
(Existing `idx_skill_uses_agent` already covers this — verify; if equivalent, skip.)

### Implementation sketch
Owner: `bene/kernel/trust.py` (single module change, single method body).

Dataflow:
1. `outcome_reliability(agent_id)` issues two SELECTs ordered by timestamp DESC:
   - `SELECT used_at AS ts, COALESCE(quality, CAST(success AS REAL)) AS q FROM skill_uses WHERE agent_id=?`

Library is VERIFIED at bene/kernel/evolve/localize.py + tests/kernel/test_localize.py (~447 + 393 LOC); no CLI command, no experiments-journal integration, hints/coefficients are module constants (not breedable).  
**LOC estimate**: ~320 lines  
**Verdict**: ⚠️ flagged for revision

### Scene
An adx-cli agent fails its 5th `mh search` iteration: the visible error is a `tool:reload-cache-service` boom on step 9, but the agent actually drifted three steps earlier when it logged `intent: switch cache backend to lru`. The reviewer runs `bene localize <agent_id> --since 2h` and gets the intent engram id, a 0.78 confidence rationale, and a tier-1 episodic engram persisted with `derived_from` lineage — pasteable straight into the post-mortem.

### Problem
When a `bene run` fails, the user gets the visible exception but no pointer to the EARLIEST decisive step that locked it in — they have to grep traces by hand. ### Proposed API

## 7. Continual Harness — in-episode mutation gated by the same kill-gate

**Status**: CLAIMS-AUDIT.md:35 — "In-episode/continual harness mutation (Continual Harness) | planned | UNVERIFIED | between-generation only today". Status: planned/new. Today `ReflectiveEvolver.run()` (`bene/kernel/evolve/gepa.py:154`) mutates strategy only between generations, outside any live agent loop; the runner (`bene/ccr/runner.py`) has no path to swap its own strategy mid-episode.  
**LOC estimate**: ~650 lines  
**Verdict**: ⚠️ flagged for revision

### Scene
Turn 184 of a Pokemon-Crystal run: retrieval_policy="recency_window" keeps re-loading the same 8 fight logs, loop-guard trips on `use_move("Tackle")` for the 5th time, error-rate signal is 0.6. Instead of dying, the runner pauses, locks a tiny in-episode probe over a 30-turn replay, mutates *only* `retrieval_policy` → `relevance_topk`, the probe says ACCEPT, the runner hot-swaps the component on turn 215 and resumes — provenance, lock hash, verdict, and the swap are all engrams that show up in `bene experiments ls`.

### Problem
Long-horizon agents (>200 turns) currently can't change strategy when their genome demonstrably stops working — they grind to context-pollution / loop-guard trip / cost blowout and have to die and respawn before any mutation ships. We need a way for a running agent to swap *one* component of its active genome mid-episode without ever bypassing the ACCEPT verdict requirement.

### Proposed API

```python
# bene/kernel/evolve/continual.py  (new module, pillar 2)
from bene.kernel.evolve.gepa import Genome, PromotionBlocked
from bene.kernel.eval.probe import Probe

class ContinualMutator:
    """In-episode mutation. Owns no policy of its own — it only
    (a) freezes a replay window, (b) builds a candidate component,
    (c) runs a *registered* probe, (d) on ACCEPT swaps the component
    in agent state under an autonomy-gated capability."""

    def __init__(self, store, conn, *, autonomy, registry,
                 reflect_fn, replay_evaluator,
                 min_turns_between_swaps: int = 50,
                 max_swaps_per_episode: int = 4): ...

    def maybe_mutate(
        self,
        agent_id: str,
        active_genome: Genome,
        trigger: dict,         # {"reason": "loop_guard"|"pollution"|"cost_spike"|"manual",
                               #  "intervention_engram_id": str|None,
                               #  "replay_window_ids": list[str]}
    ) -> "SwapDecision":       # status in {"SKIPPED","VOID","REJECTED","ACCEPTED_SWAPPED"}
        ...

    def active_genome(self, agent_id: str) -> Genome | None: ...

# capability registered in bene/kernel/capabilities.py
Capability(name="evolve.in_episode_swap", autonomy_level=3,
           description="hot-swap one genome component mid-episode after ACCEPT")

# CCR runner integration (bene/ccr/runner.py): one new optional arg
Runner(..., continual_mutator: ContinualMutator | None = None,
       continual_triggers=("loop_guard","pollution","cost_spike"))
# Wiring: in the existing loop-guard / pollution observation paths the
# runner calls `mutator.maybe_mutate(agent_id, active, trigger)`.
# On ACCEPTED_SWAPPED the runner replaces the live Genome reference for
# the next turn; on every other status it does nothing (no implicit retry).
```

```bash
# CLI (under existing `bene evolve` group, no new top-level command)
bene evolve continual status   <agent>                 # active genome + swap history
bene evolve continual swap     <agent> <component> <new_text_file> \
                               --probe <probe_name> --replay-window <N>  # manual entry
bene evolve continual disable  <agent>                 # revoke L3 grant for this cap
bene --json evolve continual status <agent>            # JSON for jq / agents
```

### Data structures
No new top-level subsystem. Adds:

1. New SQLite table:
```sql
CREATE TABLE continual_swaps (
  swap_id TEXT PRIMARY KEY,                    -- ULID
  agent_id TEXT NOT NULL,
  episode_id TEXT NOT NULL,                    -- agent run / session id
  trigger_reason TEXT NOT NULL,                -- 'loop_guard'|'pollution'|'cost_spike'|'manual'
  trigger_engram_id TEXT,                      -- intervention/pollution engram that fired the swap
  parent_genome_engram_id TEXT NOT NULL,       -- pre-swap strategic engram
  child_genome_engram_id TEXT NOT NULL,        -- post-swap strategic engram
  component TEXT NOT NULL,                     -- one of GENOME_COMPONENTS
  probe_id TEXT NOT NULL,                      -- the probe that ACCEPTed
  verdict_engram_id TEXT NOT NULL,             -- ACCEPT eval engram (gated_by target)
  swap_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%f','now'))
);
CREATE INDEX idx_continual_swaps_agent ON continual_swaps(agent_id, swap_at);
```

2. Reuses existing engram kinds — *no* new kinds. The child genome is a `strategic` engram (tier 4), `derived_from` the parent and `gated_by` the verdict — exactly the existing `promote()` invariant. The trigger gets `consolidates`-linked from the child genome's metadata for replay.

3. New `bene.yaml` block (under existing `kernel:` section):
```yaml
kernel:
  continual:
    enabled: false               # opt-in, off by default
    min_turns_between_swaps: 50
    max_swaps_per_episode: 4
    triggers: [loop_guard, pollution, cost_spike]
    replay_window_turns: 30
```

### Implementation sketch
Owner: pillar 2 (`bene/kernel/evolve/continual.py`, ~one module). Reuses, never duplicates: probe layer (pillar 1), engram ladder (substrate), autonomy + trust (pillar 4), loop-guard / pollution sensors (pillar 4). No new top-level pillar.

Dataflow per swap:
1. Runner loop already calls `LoopGuard.observe(...)` and `PollutionDetector.scan(...)` each turn. We add one new sink: when a trigger fires AND `kernel.continual.enabled`, the runner calls `ContinualMutator.maybe_mutate(agent_id, active_genome, trigger)`. No new code path inside `runner.py` beyond ~30 lines of wiring.
2. `maybe_mutate` enforces budgets: `min_turns_between_swaps`, `max_swaps_per_episode` (read from `continual_swaps`). If exceeded → return `SKIPPED`, no engram churn.
3. Build replay window: take the last N turn engrams (tier 0) from this agent, hand them to `replay_evaluator(genome) -> {quality, cost, tokens}`. This is the *same* signature as `ReflectiveEvolver.benchmark`, so we get one source of truth for "does this genome work on this trace".
4. Call `reflect_fn(active_genome, feedback)` → returns `{"component", "new_text", "rationale"}` — identical to the between-generation contract. Build `child = active_genome.mutate(component, new_text)`.
5. Run a **registered, hash-locked probe** that was registered at episode start (or via `bene evolve continual swap --probe <name>`). Subject = `child` evaluated on the replay window; baseline = `active_genome` evaluated on the same window. Probe is the *normal* `Probe` class from pillar 1 — no parallel "lite" gate. If unregistered → return `SKIPPED` (no implicit registration; that would be a back door).
6. On `REJECT` or `VOID`: the swap is rejected, child genome is *not* persisted (saves engram churn on the noisy path), but the verdict engram and an `intervention`-kind "rejected_swap" engram are written for audit.
7. On `ACCEPT`: dispatch through `CapabilityRegistry` for `evolve.in_episode_swap` (autonomy L3 default). If denied → `AutonomyDenied`, trust engram logged via the existing `AutonomyPolicy.check` path. If allowed → persist child genome as `strategic` engram, `link(child, parent, "derived_from")`, call existing `evolve.promote(child_engram_id)` → emits the `gated_by` link to the verdict. **`promote()` is unchanged** — we go through the front door. Insert row into `continual_swaps`. Set agent state `active_genome_engram_id = child_engram_id`. Next runner turn reads the new genome reference.
8. Trust signal: each ACCEPTED_SWAPPED contributes to `outcome_reliability`; each REJECT contributes negatively. Reuses existing `TrustLedger` write paths — no new signal.

Hard contracts preserved:
- L4 is never auto-granted. The capability is L3; mass strategy promotion across agents still needs an explicit human L4 grant elsewhere.
- Probes are sha256-locked at registration in the probe layer; we register them once, run them many times. We never edit a probe spec to make it pass — REJECT stands.
- `promote()` is the only path that flips a candidate to live. Continual reuses it.
- Provenance is mandatory; the child engram has `derived_from` parent + `gated_by` verdict + `metadata.trigger_engram_id`.

### Scope boundary
v0.3 ContinualMutator does NOT: (a) mutate more than one component per swap (multi-component swaps wait until per-component credit assignment lands in continual feedback), (b) auto-register probes — if no probe is registered for `continual-swap` the trigger returns SKIPPED, never invents a gate, (c) cross-agent swap propagation — a swap is scoped to one agent; broadcasting accepted genomes across the fleet stays in the between-generation `mh_search` path. ### Risks
- Reward-hack on the replay window: a child genome over-fits the 30 turns used as both replay and probe baseline, then regresses post-swap; mitigated by requiring the replay window to come from BEFORE the trigger turn (held-out future turns), and by adding a follow-up `outcome_reliability` trust signal that surfaces 'accepted-then-regressed' agents.
- Swap thrash under noisy triggers: a flaky loop-guard signal could fire `maybe_mutate` every 50 turns and burn cost on rejected probes; mitigated by `max_swaps_per_episode=4` default plus exponential backoff on consecutive REJECTs.
- Hidden state coupling — if the child genome's new `tool_config` references tools the agent loop has already partially consumed (e.g., open file handles, retrieval cache), a mid-episode swap can leave dangling state; v0.3 only swaps `prompt`, `retrieval_policy`, `context_strategy` components by default (the three pure-policy components), with `tool_config` and `memory_policy` gated behind an explicit `--allow-stateful` flag.
- Performance: each trigger now incurs a replay-window benchmark (N turns of `replay_evaluator`); on a 200-turn episode with 4 triggers that's potentially 4×30 = 120 extra evaluator calls; mitigated by per-agent budget cap and by reusing engram metadata caches (no recomputation if the same window was scored in the last 10 turns).

**Status**: CLAIMS-AUDIT line 27 (Skill plasticity row) = **PARTIAL** — outcome-weighted retrieval + continuous quality is already VERIFIED (line 28/29, `bene/skills.py` + `tests/test_skills_weighted.py`, Round-3 closes A1b/A2/A5). Demotion/retirement policy is still PLANNED (BENE-4, BENE2-DESIGN.md L63). Status: ship the missing plasticity half; do not retouch the verified ranking math.  
**LOC estimate**: ~760 lines  
**Verdict**: ⚠️ flagged for revision

### Scene
Day 9 of a 14-day benchmark sweep. `agent-research-03` has run `bene skills search "classification ensemble" --rank weighted` 41 times today; skill #17 `ensemble_classifier_v1` keeps surfacing in the top-3 even though its last 12 `record_outcome` rows in `skill_uses` are all `success=0, quality=0.1` (a sibling library bumped its API and v1 silently broke). The operator wants `bene skills plasticity scan` to mark #17 `demoted` automatically, link it `supersedes` to the v2 successor when one exists, and have `search(rank="weighted")` quietly drop demoted skills from results — without ever deleting the row or touching the verified BM25 × Wilson × recency math.

### Problem
A skill that used to work (4× successes a month ago) but has failed its last 8 uses still ranks high by Wilson-LB because the recency decay only adjusts the *retrieval order* — it never takes the skill off the menu or marks the supersession. Agents keep picking known-broken templates, wasting tokens and producing flaky outputs, and operators have no auditable record of *why* a skill was demoted.

### Proposed API

```python
# bene/kernel/memory/plasticity.py  (new module — owns the policy)

from dataclasses import dataclass
from bene.kernel.eval.probe import Probe  # plasticity probe is sha256-locked

@dataclass(frozen=True)
class PlasticityPolicy:
    min_uses: int = 8                 # below this, never demote (cold-start protection)
    recent_window: int = 10           # tail length scanned in skill_uses
    failure_threshold: float = 0.25   # mean(quality) over window; <= this triggers demotion
    half_life_days: float = 14.0      # matches DEFAULT_HALF_LIFE_DAYS in skills.py
    retire_after_days: float = 90.0   # demoted + idle this long → retired
    require_probe_accept: bool = True # promotions/demotions go through the kill gate

class PlasticityScanner:
    def __init__(self, conn, store, probe: Probe | None = None): ...

    def scan(self, *, dry_run: bool = False,
             now: datetime | None = None) -> "PlasticityRun":
        """Inspect every skill with >= min_uses; emit Demote / Retire / Hold
        decisions; persist demotion engrams + skill_lifecycle rows (commit'd
        only when the kill-gate probe ACCEPTs and dry_run is False)."""

    def supersede(self, old_skill_id: int, new_skill_id: int,
                  *, verdict_engram_id: str) -> str:
        """Record supersedes link old→new in engram_links; requires an ACCEPT
        verdict engram (same contract as evolve.promote). Returns new
        engram link_id. Raises PromotionBlocked otherwise."""

# CLI surface (extends existing `skills` group; no new top-level group)
$ bene skills plasticity scan --dry-run [--policy plasticity.yaml]
$ bene skills plasticity scan                # commits demotions through probe
$ bene skills plasticity supersede 17 42 --verdict <engram_id>
$ bene skills ls --include-demoted           # default hides demoted/retired
$ bene skills lifecycle 17                   # show transitions + reasons
```

`SkillStore.search(rank="weighted")` gets one minimal patch: a default filter `WHERE NOT EXISTS (SELECT 1 FROM skill_lifecycle l WHERE l.skill_id = s.skill_id AND l.status IN ('demoted','retired'))`, bypassable via new arg `include_demoted=False`. The BM25 × Wilson × recency math stays byte-identical.

### Data structures
**New table** `skill_lifecycle` (additive; lives in `bene/schema.py` MIGRATION_V5_SQL, never alters `agent_skills`):
```sql
CREATE TABLE IF NOT EXISTS skill_lifecycle (
    lifecycle_id   INTEGER PRIMARY KEY AUTOINCREMENT,
    skill_id       INTEGER NOT NULL REFERENCES agent_skills(skill_id),
    status         TEXT NOT NULL CHECK (status IN
                   ('active','demoted','retired','restored')),
    reason         TEXT NOT NULL,            -- e.g. "mean_quality=0.08 over last 10 uses"
    decided_by     TEXT NOT NULL,            -- "policy:PlasticityScanner" or "human:<name>"
    verdict_engram TEXT,                     -- engram_id of the ACCEPT verdict (nullable for human override)
    superseded_by  INTEGER REFERENCES agent_skills(skill_id),
    decided_at     TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%f','now'))
);
CREATE INDEX idx_skill_lifecycle_skill  ON skill_lifecycle(skill_id, decided_at DESC);
CREATE INDEX idx_skill_lifecycle_status ON skill_lifecycle(status, decided_at DESC);
```
Latest-row-per-skill = current status; never UPDATE existing rows — append-only audit trail, matches the engram discipline.

**New engram kind reuse** (no schema change): demotion decisions are written as `kind='intervention'` engrams with `metadata={"plasticity": True, "skill_id": N, "from": "active", "to": "demoted", "policy_hash": "..."}`. Supersedes between skills reuse the existing `engram_links.link_type='supersedes'` (already in v2 schema L43).

**New config section** in `bene.yaml`:
```yaml
plasticity:
  enabled: true
  min_uses: 8
  recent_window: 10
  failure_threshold: 0.25
  retire_after_days: 90
```

No engram-kind expansion needed; reuses `intervention` (already in CHECK constraint L19-22 of schema_v2).

### Implementation sketch
**Owning module:** `bene/kernel/memory/plasticity.py` — new, sits next to `lifecycle.py` (memory granule consolidation) since both are "scheduled maintenance against engrams". Fits Subsystem 3 (Context OS / memory) per BENE2-DESIGN §Pillar 1.

**Dataflow per scan:**
1. `PlasticityScanner.scan()` reads `agent_skills` × `skill_uses` (tail-N per skill, no full table scan — uses the existing `idx_skill_uses_skill(skill_id, used_at)` index).
2. For each skill with `use_count >= min_uses`: compute `mean_quality` over the last `recent_window` rows (NULL quality → `success` proxy, same coercion `_quality_signal_map` already does).
3. Decision tree: `mean_quality <= failure_threshold` ⇒ `Demote`; current `demoted` AND idle ≥ `retire_after_days` ⇒ `Retire`; else `Hold`.
4. For each non-Hold decision, build a sha256-locked `Probe` (Subsystem 2 / eval) with one gate: `"current_window_mean_quality > demoted_window_mean_quality"`. The probe's *baseline* is the skill's pre-demotion 30-day rolling window; *subject* is the current `recent_window`. If `mean(subject) <= failure_threshold` AND `mean(subject) < mean(baseline)`, the gate kills → REJECT means "the failing-skill claim is verifiable" → demotion goes through. (Inverting verdicts to fit the kill-gate semantics: ACCEPT = "no degradation observable", so we demote only on REJECT.)
5. `_record_demotion` writes (a) `intervention` engram with policy hash + decision, (b) `skill_lifecycle` row, (c) `engram_links` `gated_by` link → the verdict engram, mirroring `evolve.promote()`'s pattern exactly.
6. The `skills.SkillStore.search` weighted-mode SQL gets a single LEFT JOIN against `skill_lifecycle` (latest row per skill via window function or correlated subquery) and filters `status NOT IN ('demoted','retired')` unless `include_demoted=True`.

**Integration with the 5 subsystems:**
- **Engram ladder:** every decision = one `intervention` engram, queryable through `engram_fts`. No new kind.
- **Probe + kill gate:** demotion goes through `Probe.run()` exactly like `evolve.promote()`; no back-door. Probe spec is sha256-locked at first scan and reused.
- **Context OS:** `plasticity.py` lives in `bene/kernel/memory/` next to `lifecycle.py` (granule consolidation) — same "scheduled maintenance" pattern, same idempotent `run/plan` shape.
- **Autonomy + trust:** scanner runs at L2 (auto-execute reversible policy). `supersede()` (which actually rewires which skill agents see) requires `granted_by="human:<name>"` if the policy config says `require_human_supersede: true`; default policy demotes auto but never supersedes auto. L4 hard contract preserved.
- **Breeding:** when `evolve.distill.py` produces a successor skill, it can call `PlasticityScanner.supersede(old, new, verdict_engram_id=accept_verdict)` — same kill-gate plumbing as `promote()`.

**No mutation of source rows:** demotion never touches `agent_skills.use_count/success_count/template/etc`. Restoring a skill = appending a `status='restored'` row to `skill_lifecycle`. The base table is immutable to plasticity.

### Risks

- Recency-decay double-counting: the search-time recency multiplier already discounts old failures; if plasticity also uses the same half-life it may over-penalize. Mitigation = scan uses a fixed `recent_window` row count, not a time-decayed weight, so the two signals stay orthogonal.
- Probe lock churn: if `PlasticityPolicy` is tuned (e.g. `failure_threshold` 0.25 → 0.3), the probe spec changes → automatic REJECT on next run. Acceptable cost (forces a re-register + named version bump), but operators must know — document in `bene skills plasticity scan --help`.
- Migration: `skill_lifecycle` table is new; databases that ran an older BENE will get the table on next open via the additive migration, but if a user opened the DB read-only the migration silently no-ops and `scan` will raise OperationalError.
